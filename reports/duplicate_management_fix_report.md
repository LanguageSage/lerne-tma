## 1. Архитектура и принцип работы системы определения дубликатов

Система управления дубликатами в Lerne TMA предназначена для автоматического выявления, группировки и удобного разрешения конфликтов между повторяющимися карточками пользователя. Ниже подробно описано, как функционирует каждый уровень этой системы.

### 1.1. Серверный уровень (Бэкенд)
Определение дубликатов производится на уровне базы данных SQLite (через ORM Peewee) в сервисе `api/services/cards.py`. Процесс разделен на два этапа для обеспечения максимальной производительности:

1. **Поиск дублирующихся текстов (`duplicate_texts_query`)**:
   Сервер выполняет SQL-запрос с группировкой по полю `front_text` среди всех активных (`is_deleted == False`) карточек пользователя (связанных через таблицу `TMA_Deck` с `user_id`). С помощью условия `HAVING COUNT(id) > 1` отбираются только те формулировки, которые встречаются в базе данных более одного раза.

2. **Извлечение полных данных карточек (`all_duplicates`)**:
   Получив список повторяющихся текстов (`text_list`), сервер выполняет второй запрос, извлекающий все карточки, чей `front_text` входит в этот список. При этом карточки сортируются по алфавиту (`order_by(TMA_Card.front_text)`), чтобы связанные дубликаты шли друг за другом.
   Для каждой карточки собирается полный набор метаданных и медиафайлов: `id`, `deck_id`, `front`, `back`, `context`, `image_path`, `audio_path`, `audio_back_path`, `video_front_path`, `video_back_path`, `want_to_learn` и название колоды `deck_name`.

3. **Выборка для режима изучения (`get_next_duplicate_card`)**:
   Для последовательного просмотра дубликатов в режиме карточек используется аналогичный механизм фильтрации, который дополнительно учитывает список исключений (`exclude_ids`), чтобы не показывать одну и ту же карточку повторно в рамках одной сессии, и автоматически создает/обновляет прогресс изучения (`TMAProgress`).

### 1.2. Клиентский уровень (Фронтенд)
На клиентской стороне (React + Zustand) работа с дубликатами организована через специализированное хранилище и компоненты:

1. **Глобальное хранилище (`useDeckStore.js`)**:
   Хранит массив `duplicateCards` (загружаемый через эндпоинт `/cards/duplicates`) и идентификатор `lastDuplicateCardId` для отслеживания последней просмотренной карточки.

2. **Группировка и отображение (`DuplicateManager.jsx`)**:
   Полученный плоский массив дубликатов динамически группируется на фронтенде с помощью `reduce` по ключу `card.front`. Это позволяет визуально объединять пары, тройки и более копий одной и той же фразы в наглядные карточки-группы (`cards-stack`). Пользователь видит точное количество найденных карточек и количество уникальных групп.

3. **Интерактивное управление**:
   Внутри каждой группы пользователь может совершать действия:
   - **Просмотр (`onViewCard`)**: Открывает карточку в полноэкранном режиме изучения (`StudyView`) с виртуальной колодой `'duplicates'`.
   - **Редактирование (`onEdit`)**: Открывает карточку в редакторе (`useCardEditor`), позволяя изменить текст, контекст или прикрепленные медиафайлы.
   - **Удаление (`onDelete`)**: Безвозвратно удаляет лишнюю копию карточки с мгновенным обновлением списка.

## 2. Цели и задачи выполненных работ
Основной целью недавних улучшений являлось устранение комплекса ошибок и улучшение пользовательского опыта (UX/UI) при работе со списком дубликатов, а также оптимизация процессов удаления и стабильности БД, а именно:
1. Устранение ошибок при редактировании и сохранении карточек-дубликатов.
2. Отключение системной блокировки на создание и редактирование дублирующих карточек.
3. Исправление логики навигации (кнопки влево/вправо) при просмотре дубликатов в стандартном режиме и в режиме автовоспроизведения.
4. Реализация надежного сохранения и автоматического восстановления позиции скролла при возврате из карточки в общий список дубликатов.
5. Устранение избыточного (двойного) подтверждения при удалении карточек.
6. Устранение критической ошибки исчерпания пула соединений базы данных (`Exceeded maximum connections`).

## 3. Выполненные архитектурные и функциональные изменения

### 3.1. Сервисы и архитектура бэкенда (`api/main.py`, `api/services/cards.py`)
- **Управление пулом соединений (`api/main.py`)**: Внедрен официальный Starlette-мидлвар `@app.middleware("http") async def db_session_middleware`, который гарантирует автоматическое закрытие и возврат соединений в пул Peewee (`tma_db.close()`, `lerne_db.close()`) после завершения каждого HTTP-запроса. Это полностью устранило критическую ошибку `Exceeded maximum connections`, из-за которой сервер переставал отвечать на запросы, возвращал пустые списки дубликатов и приводил к внезапному исчезновению карточки дубликатов с главного экрана.
- **Расширение выборки данных**: В функции `get_duplicate_cards` добавлена полная выборка полей карточки (`context`, `image_path`, `audio_path`, `audio_back_path`, `video_front_path`, `video_back_path`, `want_to_learn`, `deck_id`). Это гарантирует сохранение всех прикрепленных медиафайлов и метаданных при переходе в режим редактирования.
- **Снятие ограничений на дубликаты**: Полностью удален валидационный блок в функции `save_card`, который ранее блокировал сохранение карточек при совпадении `front_text` и `back_text` в рамках одной колоды (ошибка `Такая карточка уже есть в этой колоде`). Система теперь полностью разрешает создание и редактирование дубликатов.
- **Защита от некорректных ID колод (`invalid literal for int()`)**: В функции `save_card` реализован безопасный парсинг `deck_id`. Если с клиента приходит строковый идентификатор режима (например, `'duplicates'`), бэкенд корректно обрабатывает его, не допуская сбоя базы данных и сохраняя карточку в её исходной колоде.
- **Обогащение данных для фронтенда**: В утилиту `_build_card_dict` добавлен возврат поля `"deck_id"`, чтобы клиентская часть всегда имела доступ к реальному идентификатору колоды при просмотре карточки.

### 3.2. Управление глобальным состоянием (`app/src/store/useDeckStore.js`)
- Добавлено новое поле `lastDuplicateCardId: null` и соответствующий метод `setLastDuplicateCardId`. Данное состояние отвечает за хранение идентификатора последней карточки, которую открывал пользователь из списка дубликатов.

### 3.3. Клиентская логика и хуки
- **`useCardEditor.js`**: 
  - Настроена корректная подготовка поля `deck_id` перед отправкой на сервер. Если карточка редактируется из раздела дубликатов, вместо виртуального ID `'duplicates'` подставляется реальный `deck_id` карточки.
  - Добавлена логика синхронизации: при успешном сохранении изменений в карточке из режима изучения дубликатов автоматически вызывается `fetchDuplicates()`, благодаря чему пользователь, возвращаясь в список дубликатов, сразу видит обновленные данные.
  - В метод `handleDeleteCard` добавлен параметр `skipConfirm = false`. Это позволяет отключать встроенный `window.confirm`, предотвращая появление второго всплывающего окна, если подтверждение уже было получено от пользователя.
- **`useStudySession.js`**: 
  - Полностью переработан механизм выбора следующей карточки (`fetchNextCard`) для виртуальной колоды `'duplicates'`. Вместо фильтрации по истории изучения (которая приводила к сбросу навигации) теперь используется строгий поиск по индексу текущей карточки в массиве `duplicateCards`.

### 3.4. Компоненты пользовательского интерфейса
- **`StudyView.jsx`**: 
  - В обработчиках кнопок навигации для режима автовоспроизведения (`handleAutoplayAwareBack` и `handleAutoplayAwareNext`) внедрен динамический выбор источника карточек (`const cards = currentDeck?.id === 'duplicates' ? duplicateCards : deckCards;`). Это полностью восстановило работоспособность стрелок переключения в режиме автовоспроизведения дубликатов.
  - При нажатии на кнопку "Назад" в шапке карточки фиксируется её ID в глобальном хранилище (`setLastDuplicateCardId`).
  - При нажатии на кнопку редактирования (`onOpenEditor`) передается подлинный `card.deck_id`.
  - При удалении дубликата (`onDeleteDuplicate`) в `handleDeleteCard` передается флаг `skipConfirm: true`.
- **`DuplicateManager.jsx`**: 
  - Каждому DOM-элементу карточки в списке добавлен уникальный атрибут `id={`duplicate-card-${card.id}`}`.
  - При клике на карточку (`onViewCard`) её ID сохраняется в `useDeckStore`.
  - Внедрен хук `useEffect`, который при монтировании компонента проверяет наличие сохраненного `lastDuplicateCardId` и с задержкой `setTimeout(..., 150)` (после полной отрисовки DOM) плавно прокручивает список к выбранной карточке (`el.scrollIntoView({ behavior: 'smooth', block: 'center' })`).
  - При удалении карточки (`onDelete`) в `handleDeleteCard` передается `skipConfirm: true`.
- **`App.jsx`**: 
  - При удалении карточки из модального окна действий (`CardActionModal`) в `handleDeleteCard` передается `skipConfirm: true`.
  - В `useEffect`, отвечающем за сброс скролла (`root.scrollTo({ top: 0 })`), добавлено исключение: если текущий вид — `duplicates` и в хранилище есть `lastDuplicateCardId`, сброс скролла в 0 блокируется, позволяя компоненту `DuplicateManager` беспрепятственно прокрутить экран к нужной карточке.

## 4. Итоги и результаты
1. **Пул соединений БД**: Устранена утечка соединений базы данных. Сервер работает стабильно, не выбрасывая ошибок `Exceeded maximum connections`, и колода дубликатов корректно отображается на главном экране.
2. **Стабильность сохранения**: Карточки-дубликаты успешно сохраняются и редактируются без серверных ошибок и без сбоев валидации `int()`.
3. **Свобода управления контентом**: Пользователи могут беспрепятственно создавать и изменять дублирующие карточки.
4. **Безупречная навигация**: Переключение между карточками-дубликатами работает предсказуемо и последовательно в любых режимах (включая автовоспроизведение).
5. **Премиальный UX**: Позиция скролла в списке дубликатов надежно сохраняется и автоматически восстанавливается (с плавной прокруткой к карточке) при возврате из просмотра, так как устранен конфликт с глобальным сбросом скролла.
6. **Комфортное удаление**: Устранено раздражающее двойное подтверждение — удаление карточки или колоды теперь требует ровно одного клика для подтверждения.
